## Discussion: Normalized Elevation Values and Z Factor Relationship

**Question:** What is the relationship between the normalized heightmap values and the hillshade Z Factor when interpreting the image as a topographic map?

**Explanation:**

The heightmap loader normalizes all pixel values to the 0.0–1.0 range (pixel brightness / bit-depth maximum). The hillshade algorithm computes terrain slope as `dz/dx` where `dz` is in normalized units and `dx` is in pixels. The Z Factor scales `dz` to make the slope geometrically accurate.

For a geographically correct hillshade:
```
Z Factor = elevation_range_in_real_units / ground_resolution_in_same_units_per_pixel
```

Example: A 1000×1000px image covering 10km × 10km with 1500m elevation range:
- Ground resolution = 10,000m / 1000px = 10 m/px
- Correct Z Factor = 1500 / 10 = 150

This is exactly the app's default Z Factor — not a coincidence for typical terrain data.

**Potential feature:** Add a "Ground Resolution" field (real-world units per pixel, or total map width/height). With elevation calibration already present, the app could auto-suggest the geometrically correct Z Factor and display it as a read-only hint next to the slider. Unit conversion would follow the existing `calToMeters` / `calFromMeters` pattern.

User acknowledged the explanation; no implementation decision made yet.
