## Framing Section Position + Export Frame Support

Two follow-up changes to the Phase 1 frame shell.

### Framing section moved to bottom of ParameterPanel

The Framing section was originally placed between Hillshade and Contour Parameters.
Moved it to the very bottom of the panel, after the Sea Level collapsible section,
with a `<Divider />` separating it from Sea Level above.

### Frame included in all export paths (`src/renderer/src/utils/export.ts`)

Both `ExportLayerConfig` and `OverlayExportConfig` gained two optional fields:
- `frame?: FrameConfig` — the frame configuration
- `includeFrame?: boolean` — whether to composite it into the exported image

When `includeFrame && frame`:
- The export canvas is expanded by the four margin amounts
- The margin area is filled with `frame.marginColor`
- All map content (base image, contour SVG, annotation SVG, grid) is drawn at
  `(marginLeft, marginTop)` within the larger canvas
- `drawFrameBorder()` renders the chosen border style on top using canvas APIs
  (mirrors the five CSS styles: single strokeRect, double strokeRect ×2,
  cartographic solid+dashed, shadow with canvas shadow, ornate double+corner squares)

**Grid confinement:** `drawSquareGrid` and `drawHexGrid` now accept `offsetX, offsetY`
parameters. Both draw to a temp canvas sized for the map area only, then composite at
the map offset — making it impossible for grid lines to appear in the frame margin area,
regardless of the grid pattern or tile size.

### Overlay export modal — "Include frame" switch

`OverlayExportModal` receives the full `FrameConfig` as a prop. When `frame.enabled`,
a "Frame" section with an "Include frame in export" switch (default on) appears at the
bottom of the modal before the Export button.

### Standard exports — "Export options" modal

The four non-modal exports (merged terrain, merged hillshade, unmarked hillshade,
visible map) now route through `requestExport()` in Toolbar:
- If `frame.enabled`: opens a small "Export options" modal with the frame toggle,
  then runs the export with the chosen setting on confirm
- If `!frame.enabled`: runs the export immediately (no modal, preserving existing UX)

## Export Resolution Discussion + Community/Premium Feature Planning

### Export resolution cap (not yet fixed)

Questioned why exports are limited to ~1000×1000px when heightmaps can be much larger.

**Root cause:** Both `exportToBlob` and `exportOverlayToBlob` source canvas dimensions
from `getBoundingClientRect()` on the live SVG element — this returns screen CSS pixels,
not heightmap pixels. The heightmap's native resolution is available in the store
(`heightmap.width`, `heightmap.height`) and the SVG viewBox already uses those
coordinates; nothing in the rendering pipeline prevents full-resolution output. The fix
is to pass `mapW = heightmap.width` / `mapH = heightmap.height` to the export functions
instead of reading from the DOM.

Decision: defer the fix; log it as a premium-tier feature instead.

### community-premium-features.md (`docs/community-premium-features.md`)

New document created to track premium feature candidates. Initial entry:

- **High-Resolution Export** — export at heightmap native resolution rather than screen
  resolution. Affected files when implemented: `export.ts`, `Toolbar.tsx`,
  `OverlayExportModal.tsx`.

**Commits:** `d473e4f`, `cb71bc3`

### Build tier system (documented, not yet implemented)

Documented the recommended approach for community vs. premium builds:

- **Mechanism:** Vite `define` option — set `__PREMIUM__: true/false` based on
  `process.env.BUILD_TIER`. At build time the constant is folded and dead branches are
  tree-shaken, so premium code is physically absent from the community bundle.
- **TypeScript:** `declare const __PREMIUM__: boolean` ambient declaration in `env.d.ts`.
- **Scripts:** `dev:premium` / `build:premium` via `cross-env BUILD_TIER=premium`;
  plain `dev` / `build` produce the community tier.
- **Guard syntax:** `if (__PREMIUM__) { … }` in source; UI teaser elements can use
  conditional renders instead of outright removal.

Full implementation detail in `docs/community-premium-features.md`.

- Per arm: shaft line, optional filled arrowhead triangle, label text

- Called in both `exportToBlob` and `exportOverlayToBlob` after `drawTitle`,

`compass` added to `ExportLayerConfig` and `OverlayExportConfig`, threaded through

## Compass Rose — Style Variants

Added 5 selectable compass rose styles, replacing the single plain style.

### Type change (`src/renderer/src/types/index.ts`)

```typescript
export type CompassStyle = 'plain' | 'compass' | 'nautical' | 'celtic' | 'dragon'

// Added to CompassConfig:
compassStyle: CompassStyle  // default: 'plain'
```

### ParameterPanel — Style selector

A `Select` dropdown for Style added immediately below the "Show compass" switch.
Arrow toggles in the Labels section are conditionally hidden when `compassStyle !== 'plain'`
since the shape itself conveys direction for all other styles.

### Styles (SVG live preview + canvas export)

All styles share the same coordinate system: center at origin, N arm toward (0, −size),
implemented as a `<g transform="translate(cx, cy)">` in SVG and `ctx.translate(cx, cy)`
on canvas. Labels use a shared `CardinalLabels` / `cardinalLabels` helper.

**Plain** — unchanged: 4 line arms with optional filled arrowhead triangles.

**Compass Star** — two overlapping 4-pointed stars:
- Intercardinal star (rotated 45°, scale 0.65): thin 4-pointed diamond star
- Cardinal star (scale 1.0): thin 4-pointed diamond star on top
- Inner radius for star waist: `size * 0.22`
- N arm drawn as a separate white-filled diamond polygon, outline in compass color —
  makes North visually distinct at a glance

**Nautical** — 16-point windrose:
- 4 cardinal diamond points (N/S/E/W): `tipLen = size`, `halfWidth = size * 0.18`
- 4 intercardinal diamond points (NE etc.): scale 0.72, halfWidth 0.13
- 8 sub-cardinal triangular points (NNE etc. at 22.5°): scale 0.48, halfWidth 0.06
- Inner decorative ring: `<circle>` stroke at radius `size * 0.22`
- N arm white-filled (same technique as Compass Star)
- Fleur-de-lis at N tip: 3 radiating stroke lines (center spike + 2 wings) + horizontal
  collar bar, all at `lw * 0.9` stroke weight
- N label offset pushed beyond the fleur-de-lis extension (`nExtra = fr * 0.8`)

**Celtic Knot** — Celtic cross with groove-ribbon effect:
- N-S and E-W arm bands: filled `<rect>` with `rx={ah}` rounded ends, width `size * 0.22`
- Ring: `<circle>` stroke with strokeWidth matching arm band width, radius `size * 0.40`
- Groove channels: thin white strokes at 55% opacity along arm centers (between ring and
  tip) and along the ring center, creating a 3D ribbon/tube appearance
- Terminal knot circles at each arm tip: filled circle + inner white ring groove
- Canvas note: `ctx.globalAlpha = 0.55` used for groove layer, restored after

**Norse Dragon** — Vegvisir-inspired 8 staves:
- 4 cardinal staves (N/S/E/W): full arm length, 2 perpendicular cross-branches at 60%
  and 80% along the stave
- 4 intercardinal staves (NE etc.): 76% arm length, 1 perpendicular branch at 65%
- Fork terminal (dragon tail) at every stave tip: two lines diverging ±30° from the stave
  direction using rotation formula `(dx·cosF ∓ dy·sinF, dx·sinF ± dy·cosF)`
- Center decoration: runic ring (stroked circle) + center dot
- Labels at cardinal tips, pushed beyond fork extension (`gap + size * 0.14`)

### Architecture

Each style is a standalone React component / canvas function with signature
`(props: RoseProps) → JSX.Element` / `(ctx, s, color, lw, gap, fs, compass) → void`.
`CompassRoseSvg` and `drawCompassRose` dispatch to the appropriate style via a
`switch` statement.

`legend: LegendConfig` added to `ProjectState` as a top-level field (parallel to `frame`,

### Store (`src/renderer/src/store/useStore.ts`)

- `legend: defaultLegendConfig` in initial state
- `updateLegend: (l: Partial<LegendConfig>) => void` — shallow-merges updates

### MapCanvas — live legend preview (`LegendOverlay` component)

A new `LegendOverlay` component renders inside the outer composition div when
`frame.enabled && legend.enabled && heightmap`:

- Absolutely positioned in one of four margin corners via `legend.position`
- `edgeGap = max(4, borderWidth*2 + 3)` from edge; vertically centered in margin strip:
  `bottom: marginBottom/2 - boxH/2` (or equivalent for top)
- Items filtered: `showSeaLevel` requires `style.showSeaLevel`; `showElevationFlags`/`showSlopeArrows`
  require at least one flag/arrow to exist
- SVG box with background rect (fill: marginColor, stroke: legend.color)
- Per item: sample graphic (contour line for minor/major, dashed line for sea level,
  miniature flag icon, miniature slope arrow icon) + label text

Also subscribes to `legend = useStore((s) => s.legend)` in the `MapCanvas` component body.

### ParameterPanel — Legend subsection

A "Legend" labelled `<Divider>` and controls at the bottom of the Framing `<Collapse>`,
after the Compass subsection:

- **Show legend** — `Switch`; all controls disabled when off (or frame disabled)
- **Position** — `Select` (Bottom right / Bottom left / Top right / Top left)
- **Font size / Color** — `NumberInput` (6–32) + `ColorInput`, in a `Group`
- **Items** — 5 rows (Minor contour, Major contour, Sea level, Elevation flags, Slope arrows),
  each with:
  - `Switch` for show/hide; Elevation flags and Slope arrows are auto-disabled when no data exists
  - `TextInput` for the editable label; disabled when item is toggled off

### Export (`src/renderer/src/utils/export.ts`)

`ExportLayerConfig` and `OverlayExportConfig` both gained:
`legend?`, `contourStyle?`, `hasElevationFlags?`, `hasSlopeArrows?`

`drawLegend(ctx, legend, frame, contourStyle, hasElevationFlags, hasSlopeArrows, totalW, totalH)`
draws the same box on canvas:
- Background rect with legend.color stroke
- Per item: same sample graphics (canvas API mirror of the SVG)
- Label text using `ctx.measureText()` for exact sizing (canvas has real text metrics)
- Called in both `exportOverlayToBlob` and `exportToBlob` after `drawCompassRose`,
  only when `withFrame && config.legend && config.contourStyle`

These fields are threaded through `Toolbar.tsx` (added to `frameOpts` and `OverlayExportModal` props)
and `OverlayExportModal.tsx` (added as props, forwarded to `onExport`).
