## Feature: Steps Road Type

### Overview

New road type `'steps'` added to the existing road system. Steps render as a series of short perpendicular hatch marks (crossties) evenly distributed along the spline. The spline itself is invisible except for a ghost spine that appears when the road is selected, to make editing anchor points possible.

Steps participate fully in all existing road behaviors: select, shift-select, drag anchors, delete, style, label, legend entry.

### New Utilities (`src/renderer/src/utils/spline.ts`)

**`catmullRomSamplePoints(pts, closed, samplesPerSeg?)`** — densely samples a Catmull-Rom spline into `{x,y}[]` points (30 samples per segment by default). Used internally by `stepsHatchPath` and exported for future use.

**`stepsHatchPath(pts, closed, hatchWidth, hatchSpacing)`** — produces an SVG path string of perpendicular tick marks distributed along the spline:
1. Samples the curve into dense points
2. Builds a cumulative-distance array
3. Starting at `hatchSpacing / 2`, places a tick mark every `hatchSpacing` units
4. At each tick position: interpolates position, computes tangent from adjacent samples, rotates 90° for perpendicular, emits `M x1,y1 L x2,y2`
5. Returns all tick segments as a single path string

### Type Changes (`src/renderer/src/types/index.ts`)

### Rendering (`src/renderer/src/components/MapCanvas/MapCanvas.tsx`)

**Road rendering (steps branch):**
```tsx
{isSteps ? (
  <>
    {/* Ghost spine — only when selected */}
    {isSelected && (
      <path d={centerPath} stroke={road.color} strokeWidth={sw * 0.5} fill="none"
        strokeDasharray={`${sw * 2} ${sw * 2}`} strokeLinecap="round" strokeOpacity={0.4} />
    )}
    {/* Perpendicular hatch marks */}
    <path
      d={stepsHatchPath(road.points, road.closed, tw, road.hatchSpacing ?? tw)}
      stroke={road.color} strokeWidth={sw} fill="none" strokeLinecap="round" />
  </>
) : /* ... existing isSingleLine / double-line branches */ }
```

**Hit area:** same transparent wide-stroke path as all other road types (`Math.max(tw + sw * 4, 10)`).

**Road mask:** steps uses the hatch-width extent (`trackWidth + strokeWeight * 2`) for its mask, similar to double-line roads, so it occludes correctly where roads cross.

**Drawing preview:** shows ghost spine + live hatch marks while the user is placing points.

**Label:** steps labels render via `<textPath>` along the center path, same as paved/trail roads. Label text input visible in Selected Road panel whenever a steps road is selected.

**Legend sample:** 5 evenly-spaced vertical tick marks spanning the sample width.

**Legend items array:**
```tsx
legend.showSteps && roads.some(r => r.type === 'steps')
  ? { type: 'road-steps', label: legend.stepsLabel, color: roadDefaults.stepsColor }
  : null
```

### Decisions Made

- **`trackWidth` reused as `hatchWidth`** — keeps the data model minimal; the semantic is identical (a width measurement in map units).
- **`hatchSpacing` optional on `Road`** — existing saved roads without the field fall back to `road.trackWidth` at render time (`road.hatchSpacing ?? tw`), making the feature backward-compatible.
- **Ghost spine only when selected** — avoids visual clutter; the hatch marks alone are sufficient to identify the path.
- **No label for footpath, label allowed for steps** — footpath is the only type that has historically suppressed labels; steps are named features on maps (e.g. "Stone steps") so labels are useful.

## Bug: Steps Label Rendering Through Hatch Marks

### Problem

The road label for a steps road appeared along the center spine — directly through the hatch marks — making it illegible.

### Fix

Added a `dy` offset for steps in the label render: `-(tw / 2 + tw * 0.7 * 0.6)` — half the hatch width to clear the tick tips, plus ~0.42× the font size as a breathing gap. This placed the text just above the hatch endpoints.

## Feature: Per-Road Label Styling

### Overview

All road types that support labels (everything except footpath) now have per-road label style controls: font family, font size, color, opacity, label side (above/below the path), and flip (reading direction). This mirrors the controls available on curved labels.

### Data Model (`src/renderer/src/types/index.ts`)

Six new optional fields on `Road` (all backward-compatible — existing roads fall back to computed defaults):

```typescript
labelFontFamily?: string       // default: style.labelFont (global map label font)
labelFontSize?: number         // heightmap units; default: trackWidth * 0.7
labelColor?: string            // default: road.color
labelOpacity?: number          // default: 1
labelSide?: 'left' | 'right'  // left = above for a left-to-right path; default 'left'
labelFlip?: boolean            // reverses point order → flips reading direction
```

### Rendering (`src/renderer/src/components/MapCanvas/MapCanvas.tsx`)

**Key design: offset path replaces screen-Y `dy`.**

Previously, label placement used `dy` (a screen-Y shift) which worked for horizontal roads but drifted off-road for diagonal or curved paths. The new approach uses `catmullRomOffsetPath` to generate a label path that is always perpendicular to the spline at the correct clearance distance:

```typescript
const clearance = isSingleLine ? fontSize * 0.6
  : isSteps   ? tw / 2 + fontSize * 0.5
  : /* double-line */ tw / 2 + sw + fontSize * 0.5
const sideSign = (side === 'right') !== flip ? 1 : -1
// In <defs>:
d={catmullRomOffsetPath(pts, road.closed, sideSign * clearance)}
```

The `(side === 'right') !== flip` XOR mirrors the curved-label convention: flipping the point order reverses the path direction and therefore which side is "above", so the sign must be inverted to keep the label visually consistent.

The `<text>` element is simplified to `dominantBaseline="middle"` with no `dy` — the offset path carries all positioning.

**Clearance values by road type:**
| Type | Clearance from spline center |
|------|------------------------------|
| Single-line (trail) | `fontSize × 0.6` |
| Steps | `tw/2 + fontSize × 0.5` |
| Double-line (dirt/gravel/paved) | `tw/2 + sw + fontSize × 0.5` |

**`labelFlip` path reversal** is applied to the points fed into `catmullRomOffsetPath`, so the label reads in the opposite direction along the road while remaining on the correct side.

### ParameterPanel (`src/renderer/src/components/ParameterPanel/ParameterPanel.tsx`)

Label styling controls appear in the Selected Road section for all types except footpath, immediately after the Label text input:

- **Label font** — `Select` using shared `FONT_OPTIONS` (serif, sans-serif, Georgia, Times New Roman, Arial, monospace); default: global map label font
- **Label size** — `NumberInput` in heightmap units; default: `trackWidth × 0.7`
- **Label color** — `ColorInput`; default: road color
- **Label opacity** — `Slider` 0–100%
- **Label below path** — `Switch`; toggles `labelSide` between `'left'` (above) and `'right'` (below)
- **Flip reading direction** — `Switch`; toggles `labelFlip`

### Decisions Made

- **`catmullRomOffsetPath` instead of `dy`** — screen-Y `dy` fails for diagonal paths; perpendicular offset is correct for all angles and is already available in `spline.ts`.
- **Side + flip XOR** — same convention as curved labels: `(side === 'right') !== flip`. When you flip the path direction, "above" and "below" swap, so the sign must compensate to keep the user's chosen side visually consistent.
- **All fields optional** — no migration needed; existing roads use implicit defaults that produce slightly better placement than before (double-line road labels now appear above the road rather than between the tracks).
