## Feature: Unified "Recalculate Map" Button

**Motivation:** With ground resolution now feeding into hillshade Z Factor computation, changing elevation calibration can dirty both hillshade and contours simultaneously. Having two separate "Rerun Hillshading" and "Recalculate Contours" buttons forces the user to understand which subsystem is affected by which settings — a burden that grows as the settings become more interconnected.

**Design decisions (Q&A with user):**

- **Q1 (button label/behavior):** Button says "Recalculate Map" but only recalculates what is actually dirty. If only contour settings changed, only contours recalculate; if only hillshade settings changed, only hillshade recalculates; if both are dirty, both run.
- **Q2 (spinner):** Single spinner on the button, lasts until everything marked dirty is done (`hillshadeGenerating || contoursGenerating`).
- **Q3 (simultaneity):** Hillshade and contours run simultaneously when both are dirty (they already operated independently via separate effects). Neither recalculates until the button is clicked.
- **Q4 (hillshade mode):** When not in hillshade mode, hillshade controls are hidden and `hillshadeDirty` is never set. Recalculate button is disabled unless at least one dirty flag is true.

**Changes:**

### `src/renderer/src/store/useStore.ts`
- `updateHillshadeParams`: now only sets `hillshadeDirty` when `state.terrainIsHillshade` (was unconditional)
- `updateElevationCalibration`: sets `hillshadeDirty` only when `terrainIsHillshade` AND the updated fields include `realMin`, `realMax`, or `mapWidth` (the fields that affect computed Z Factor)
- `setElevationUnits`: reads `terrainIsHillshade` from `get()` at call time; sets `hillshadeDirty` if in hillshade mode

### `src/renderer/src/components/FilePanel/FilePanel.tsx`
- Removed "Rerun Hillshading" Paper section (was conditionally visible in hillshade mode)
- Removed "Recalculate Contours" Paper section
- Added unified "Recalculate Map" Paper section:
  - `disabled={!canRecalculate}` where `canRecalculate = (hillshadeDirty || contoursDirty) && !!heightmap`
  - `loading={isRecalculating}` where `isRecalculating = hillshadeGenerating || contoursGenerating`
  - `onClick` calls `triggerHillshade()` if `hillshadeDirty`, `triggerContours()` if `contoursDirty`

## Feature: Ground Resolution → Auto-Computed Z Factor + True Vertical Exaggeration

**Motivation:** The raw Z Factor slider (1–2000, default 150) has no intuitive meaning without knowing the terrain's real-world scale. The correct Z Factor for a geographically accurate hillshade is `elevation_range / ground_resolution`, where ground resolution = map width in real-world units / image width in pixels. Once this is computable, "vertical exaggeration" can be expressed as a meaningful dimensionless ratio (1× = accurate, 2× = twice as steep as reality).

**Design decisions:**
- User enters **total map width** (not ground resolution per pixel); app divides by `heightmap.width` internally
- Field label: `"Width in [abbr] of Map"` (e.g. "Width in ft of Map")
- Field disabled until Elevation Units is set
- `mapWidth` converts when switching units (same `calToMeters`/`calFromMeters` pattern as `realMin`/`realMax`/`realInterval`)
- `mapWidth` cleared when switching to custom units (ratio unknown) or clearing units
- When `mapWidth` is set and calibration is ready: Z Factor slider replaced by VE slider (0.1×–10×, default 1.0×) plus a read-only "Actual Z Factor" display
- When `mapWidth` is not set: raw Z Factor slider (1–2000) unchanged from prior behavior

**Changes:**

### `src/renderer/src/types/index.ts`
- Added `verticalExaggeration: number` to `HillshadeParameters` (default `1.0`)
- Added `mapWidth: number | null` to `ElevationCalibration` (default `null`)

### `src/renderer/src/store/useStore.ts`
- `setElevationUnits`: converts `mapWidth` alongside `realMin`/`realMax`/`realInterval`; clears `mapWidth` when switching to custom

### `src/renderer/src/hooks/useHillshade.ts`
- Added `elevationCalibration` state read + ref (same always-current ref pattern as `hillshadeParams`)
- At generation time: if `mapWidth`, `realMin`, `realMax` are all set, computes:
  ```ts
  const groundRes = cal.mapWidth / heightmap.width
  const correctZ = Math.abs(cal.realMax - cal.realMin) / groundRes
  effectiveZFactor = correctZ * params.verticalExaggeration
  ```
  Otherwise falls back to raw `params.zFactor`
- Passes `{ ...params, zFactor: effectiveZFactor }` to `generateHillshade`

### `src/renderer/src/components/ParameterPanel/ParameterPanel.tsx`
- Added `heightmap` from store (needed for `heightmap.width` in correctZFactor computation)
- Added `mapWidth` destructured from `elevationCalibration`
- Added `correctZFactor` and `hasGroundResolution` computed values:
  ```ts
  const correctZFactor = calReady && mapWidth && mapWidth > 0 && heightmap
    ? Math.abs(realMax! - realMin!) / (mapWidth / heightmap.width)
    : null
  const hasGroundResolution = correctZFactor !== null
  ```
- Added `"Width in [abbr] of Map"` NumberInput between the custom unit fields and the real-world min/max group (disabled when `!unitType`, blue border when active, dashed+dimmed when read-only)
- Updated `onClear` handler to also clear `mapWidth`
- In Hillshade section: replaced single "Vertical Exaggeration" slider with conditional display:
  - If `hasGroundResolution`: VE slider (0.1×–10×, step 0.1) + read-only "Actual Z Factor" NumberInput showing `Math.round(correctZFactor × VE)`
  - Otherwise: raw Z Factor slider (1–2000) identical to prior behavior

## Fix: Custom Unit Values Cleared Instead of Converted

**Problem:** When switching from feet/meters to custom units, all real-world values (min, max, interval, map width) were immediately cleared to null. The user had to re-enter everything in custom units manually, even when a conversion factor was available.

**Root cause:** `setElevationUnits` had an explicit branch for `newType === 'custom'` that cleared all values. The original reasoning was that the ratio wasn't known at selection time, so conversion was impossible. But the ratio is entered immediately after, making deferred conversion straightforward.

**Fix:** Three-part change using a `preCustomUnit` tracking field:

1. **Don't clear on switch-to-custom.** Instead, keep existing values and record `preCustomUnit = old.unitType` (the unit they're currently expressed in). This flag signals "values are in feet/meters, not yet converted."

2. **New `finalizeCustomConversion` store action.** Reads `preCustomUnit`, converts all four values (realMin, realMax, realInterval, mapWidth) from the source unit to the current custom unit using the standard `calToMeters` → `calFromMeters` chain, then clears `preCustomUnit` to mark conversion complete.

3. **Wire to `onBlur` of the ratio field.** Conversion fires when the user tabs out of the "N (ratio)" input with a valid value — not on every keystroke (which would convert mid-typing with a partial ratio).

**Example:** feet → custom leagues (1 league = 15840 ft):
- realMax 1500 ft → `calToMeters(1500, {feet})` = 457.2 m → `calFromMeters(457.2, {custom, base=feet, ratio=15840})` = 0.1 leagues
- mapWidth 50000 ft → 3.2 leagues
- realInterval 75 ft → max(1, round(0.005)) = 1 league

**Files:** `src/renderer/src/types/index.ts`, `src/renderer/src/store/useStore.ts`, `src/renderer/src/components/ParameterPanel/ParameterPanel.tsx`

**Note:** Subsequent changes to the ratio after initial conversion do not auto-reconvert (the user would re-enter values). This is a deliberate scope limit — the common case (first-time setup) is handled cleanly.
