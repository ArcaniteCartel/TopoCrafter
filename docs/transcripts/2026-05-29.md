## Feature: Unified "Recalculate Map" Button

**Motivation:** With ground resolution now feeding into hillshade Z Factor computation, changing elevation calibration can dirty both hillshade and contours simultaneously. Having two separate "Rerun Hillshading" and "Recalculate Contours" buttons forces the user to understand which subsystem is affected by which settings — a burden that grows as the settings become more interconnected.

**Design decisions (Q&A with user):**

- **Q1 (button label/behavior):** Button says "Recalculate Map" but only recalculates what is actually dirty. If only contour settings changed, only contours recalculate; if only hillshade settings changed, only hillshade recalculates; if both are dirty, both run.
- **Q2 (spinner):** Single spinner on the button, lasts until everything marked dirty is done (`hillshadeGenerating || contoursGenerating`).
- **Q3 (simultaneity):** Hillshade and contours run simultaneously when both are dirty (they already operated independently via separate effects). Neither recalculates until the button is clicked.
- **Q4 (hillshade mode):** When not in hillshade mode, hillshade controls are hidden and `hillshadeDirty` is never set. Recalculate button is disabled unless at least one dirty flag is true.

**Changes:**

- `updateElevationCalibration`: sets `hillshadeDirty` only when `terrainIsHillshade` AND the updated fields include `realMin`, `realMax`, or `mapWidth` (the fields that affect computed Z Factor)
- `setElevationUnits`: reads `terrainIsHillshade` from `get()` at call time; sets `hillshadeDirty` if in hillshade mode

## Feature: Ground Resolution → Auto-Computed Z Factor + True Vertical Exaggeration

**Motivation:** The raw Z Factor slider (1–2000, default 150) has no intuitive meaning without knowing the terrain's real-world scale. The correct Z Factor for a geographically accurate hillshade is `elevation_range / ground_resolution`, where ground resolution = map width in real-world units / image width in pixels. Once this is computable, "vertical exaggeration" can be expressed as a meaningful dimensionless ratio (1× = accurate, 2× = twice as steep as reality).

**Design decisions:**
- User enters **total map width** (not ground resolution per pixel); app divides by `heightmap.width` internally
- Field label: `"Width in [abbr] of Map"` (e.g. "Width in ft of Map")
- Field disabled until Elevation Units is set
- `mapWidth` converts when switching units (same `calToMeters`/`calFromMeters` pattern as `realMin`/`realMax`/`realInterval`)
- `mapWidth` cleared when switching to custom units (ratio unknown) or clearing units
- When `mapWidth` is set and calibration is ready: Z Factor slider replaced by VE slider (0.1×–10×, default 1.0×) plus a read-only "Actual Z Factor" display
- When `mapWidth` is not set: raw Z Factor slider (1–2000) unchanged from prior behavior

**Changes:**

### `src/renderer/src/types/index.ts`
- Added `verticalExaggeration: number` to `HillshadeParameters` (default `1.0`)
- Added `mapWidth: number | null` to `ElevationCalibration` (default `null`)

### `src/renderer/src/store/useStore.ts`
- `setElevationUnits`: converts `mapWidth` alongside `realMin`/`realMax`/`realInterval`; clears `mapWidth` when switching to custom

### `src/renderer/src/hooks/useHillshade.ts`
- Added `elevationCalibration` state read + ref (same always-current ref pattern as `hillshadeParams`)
- At generation time: if `mapWidth`, `realMin`, `realMax` are all set, computes:

  Otherwise falls back to raw `params.zFactor`
- Passes `{ ...params, zFactor: effectiveZFactor }` to `generateHillshade`

### `src/renderer/src/components/ParameterPanel/ParameterPanel.tsx`


  - Otherwise: raw Z Factor slider (1–2000) identical to prior behavior

## Fix: Custom Unit Values Cleared Instead of Converted

**Problem:** When switching from feet/meters to custom units, all real-world values (min, max, interval, map width) were immediately cleared to null. The user had to re-enter everything in custom units manually, even when a conversion factor was available.

**Root cause:** `setElevationUnits` had an explicit branch for `newType === 'custom'` that cleared all values. The original reasoning was that the ratio wasn't known at selection time, so conversion was impossible. But the ratio is entered immediately after, making deferred conversion straightforward.

**Fix:** Three-part change using a `preCustomUnit` tracking field:

1. **Don't clear on switch-to-custom.** Instead, keep existing values and record `preCustomUnit = old.unitType` (the unit they're currently expressed in). This flag signals "values are in feet/meters, not yet converted."

2. **New `finalizeCustomConversion` store action.** Reads `preCustomUnit`, converts all four values (realMin, realMax, realInterval, mapWidth) from the source unit to the current custom unit using the standard `calToMeters` → `calFromMeters` chain, then clears `preCustomUnit` to mark conversion complete.

3. **Wire to `onBlur` of the ratio field.** Conversion fires when the user tabs out of the "N (ratio)" input with a valid value — not on every keystroke (which would convert mid-typing with a partial ratio).

**Example:** feet → custom leagues (1 league = 15840 ft):
- realMax 1500 ft → `calToMeters(1500, {feet})` = 457.2 m → `calFromMeters(457.2, {custom, base=feet, ratio=15840})` = 0.1 leagues
- mapWidth 50000 ft → 3.2 leagues
- realInterval 75 ft → max(1, round(0.005)) = 1 league

**Files:** `src/renderer/src/types/index.ts`, `src/renderer/src/store/useStore.ts`, `src/renderer/src/components/ParameterPanel/ParameterPanel.tsx`

**Note:** Subsequent changes to the ratio after initial conversion do not auto-reconvert (the user would re-enter values). This is a deliberate scope limit — the common case (first-time setup) is handled cleanly.

## Feature: Non-Linear Shadow Depth Slider

**Problem:** The Shadow Depth (intensity) slider was linear from 0.5–10 in steps of 0.5. Most useful control is at the low end; the high end felt coarse and the range top-heavy.

**Solution:** Replace with a non-linear 37-stop slider giving finer control at low values:
- **0–1×**: steps of 0.1 (10 intervals — most granular)
- **1–2×, 2–3×, 3–4×, 4–5×**: steps of 0.25 (4 intervals each)
- **5–6×, 6–7×, 7–8×, 8–9×, 9–10×**: steps of 0.5 (2 intervals each)

**Implementation:** A linear integer-stepped Mantine `Slider` (indices 0–36) maps to a predefined `INTENSITY_STOPS` value array. The slider always moves in equal integer steps; `intensityToIndex` converts the stored float back to the nearest index on read, and `onChange` converts the index to the actual value on write.

```ts
const INTENSITY_STOPS = [
  0.0, 0.1, ..., 1.0,          // indices 0–10
  1.25, 1.5, 1.75, 2.0,        // indices 11–14
  2.25, ..., 3.0,               // indices 15–18
  3.25, ..., 4.0,               // indices 19–22
  4.25, ..., 5.0,               // indices 23–26
  5.5, 6.0, 6.5, ..., 10.0     // indices 27–36
]
```

The existing default of `intensity: 3.0` maps cleanly to index 18. Tooltip labels show whole numbers without decimals (e.g. `3×`) and fractions with just enough precision (e.g. `0.7×`, `1.25×`).

**File:** `src/renderer/src/components/ParameterPanel/ParameterPanel.tsx`

## Feature: Sea Level Contour Line

**Motivation:** When real-world elevation calibration spans negative to positive values (e.g. -200 ft to 3000 ft), there is a meaningful real-world elevation zero — sea level. It warrants a distinct contour line separate from the regular interval grid.

**Design decisions (Q&A with user):**

- **Q1 (visibility):** Toggle "Show Sea Level Contour" in a dedicated Sea Level panel section.
- **Q2 (draw order):** Rendered on top of all other contour lines.
- **Q3 (line style):** Solid / Dashed / Dotted option, own color and thickness.
- **Label:** Replace the plain "0" text label with a custom SVG icon (upward arrow + two wave curves). Icon has its own color, size, and independent show/hide toggle — not tied to the general elevation label toggle.

**Threshold computation** (inside the 300ms deferred effect, using calibration refs):
```ts
seaLevelThreshold = params.minElevation + (-cal.realMin / (cal.realMax - cal.realMin)) * normSpan
```
Maps real-world 0 into the normalized [minElevation, maxElevation] space. Only computed when `realMin < 0 && realMax > 0`.

**SVG icon** (normalized coordinate space, scaled by `seaLevelLabelFontSize`):

## Feature: Elevation Flag Tool

**Motivation:** Users need a way to annotate specific points on the map with their real-world elevation, independent of contour line spacing. A draggable, deletable flag marker with a live-updating label serves this purpose.

**Design decisions (Q&A with user):**

- **Q1 (icon style):** Golf/survey pennant-on-pole SVG icon.
- **Q2 (persistence):** Saved with the project file when project save/load is implemented; in-memory (in Zustand store) for now.
- **Q3 (export):** Flags are part of the SVG overlay and export with everything else.
- **Q4 (calibration requirement):** Flag tool is disabled (greyed out) when elevation calibration is not set; only works when real-world min/max are defined.

**Interaction model:**
- Crosshair icon to the left of TopoCrafter logo opens a `Popover` tool panel
- Elevation flag button inside the panel toggles the tool on/off (filled = active, subtle = inactive); disabled when `!calReady`
- **Tool active:** cursor becomes crosshair over map; click background → plant a flag at that pixel; Escape → cancel tool
- **Click a flag** (any mode): selects it — pole thickens, uses `majorColor`, selection ring appears at pole base
- **Click + drag a flag:** drags it to a new position; elevation label updates in real time during drag
- **Delete key:** removes the selected flag
- **Escape:** cancels the active tool and deselects

**Architecture — separate SVG for flags:**
Flags render in a second `<svg>` positioned above the contour overlay. This prevents the contour `style.opacity` from affecting flag visibility. The flag SVG uses the same `viewBox` as the contour SVG so coordinates are identical.

**Coordinate pipeline:**
```ts
// Screen → SVG coordinates (uses flagSvgRef.current.getScreenCTM())
const svgPt = new DOMPoint(clientX, clientY).matrixTransform(ctm.inverse())

// SVG coordinate → real-world elevation
const normVal = heightmap.data[py * width + px]
const elevation = Math.round(realMin + (normVal - params.minElevation) / normSpan * (realMax - realMin))
```

**Drag implementation:** Document-level `mousemove`/`mouseup` listeners (set up once with empty deps, reading latest values via refs) ensure drag continues reliably even when the cursor leaves the SVG element. React's synthetic `onMouseUp` on the flag SVG returns early when a drag is in progress — the document handler finalizes it.

**Flag SVG geometry** (in SVG coordinate space, scaled by `labelFontSize`):
- Pole: `<line>` from anchor point (fx, fy) to (fx, fy − s), `vectorEffect="non-scaling-stroke"`
- Pennant: `<polygon>` triangle at top of pole
- Label: `<text>` offset right of pole, styled by standard label parameters
- Selection ring: `<circle>` at pole base when selected

## Feature: Slope Arrow Tool

**Motivation:** Complement the elevation flag with a slope measurement tool — drop an arrow that shows the direction and angle of steepest ascent at any point on the terrain.

**Design decisions (Q&A with user):**

- **Q1 (disabled condition):** Tool disabled unless both elevation calibration AND map width (ground resolution) are set. Slope angle in degrees requires both elevation range and horizontal scale.
- **Q2 (styling):** Shares label font, bold, italic, color, and size from the existing contour label settings — same as elevation flags.
- **Q3 (live drag):** Both arrow direction and degree value update in real time as the arrow is dragged.
- **Tooltip:** "Add angle of steepest ascent to the map"

**Slope computation:**
```ts
// Central-difference gradient at pixel (px, py):
const gx = (data[py*w + x1] - data[py*w + x0]) / (x1 - x0)
const gy = (data[y1*w + px] - data[y0*w + px]) / (y1 - y0)

// Slope angle in degrees:
const gradMag = Math.sqrt(gx*gx + gy*gy)                // norm. elevation / pixel
const riseOverRun = (gradMag * |realMax - realMin|) / (mapWidth / width)
const slopeDeg = Math.round(Math.atan(riseOverRun) * 180 / Math.PI)

// Direction of steepest ascent in SVG coordinate space:
const angleDeg = Math.atan2(gy, gx) * 180 / Math.PI
```

`gy` is in SVG coordinates (Y increases downward), so `atan2(gy, gx)` correctly gives the SVG rotation angle: negative for northward slopes (arrow points up on screen), positive for southward, etc.

**Arrow geometry** (computed in absolute SVG coordinates, no SVG transform rotation):
```ts
const cos = Math.cos(angleRad), sin = Math.sin(angleRad)
// Tip and tail at ±halfLen from center (fx, fy)
// Arrowhead triangle computed with perpendicular direction (-sin, cos)
// Degree label always at (fx, fy + s*0.85) — horizontal regardless of arrow angle
```

Using inline trig rather than `<g transform="rotate()">` keeps the degree label naturally horizontal without a counter-rotation.

**Refactoring — unified selection/drag system:**
The elevation flag machinery was generalized to cover both tool types:
- `selectedFlagId` → `selectedItem: { type: 'flag' | 'slope-arrow'; id: string } | null`
- `dragRef` gains a `type` field; document `onMove`/`onUp` branch on it to call the right update action
- `DragPos` union type carries `elevation?`, `angleDeg?`, `slopeDeg?` as needed
- `handleItemMouseDown(e, type, itemId)` replaces `handleFlagMouseDown`
