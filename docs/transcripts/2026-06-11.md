## Fix: Flip / Reverse Side Rendering Logic

### Problem

The rendering logic was wired backwards:
- `side === 'right'` drove path reversal, which both changed sides AND inverted text orientation
- `flip === true` drove `dominantBaseline="hanging"`, which only changed sides (no orientation change)

Result: the two toggles together gave all four variants, but which toggle did which thing was unclear.

### Fix (two `replace_all` edits to `MapCanvas.tsx`)

**Path reversal ŌĆö now driven by `flip`:**
```
const pts = label.flip ? [...label.points].reverse() : label.points
```

**`dominantBaseline` ŌĆö XOR logic so "Reverse Side" only changes side:**
```
dominantBaseline={(label.side === 'right') !== label.flip ? 'hanging' : undefined}
```

**Verified 2├Ś2 behavior matrix:**
| side | flip | Result |
|------|------|--------|
| left | false | above path, normal reading direction |
| right | false | below path, normal reading direction |
| left | true | above path, upside-down orientation |
| right | true | below path, upside-down orientation |

## Design Discussion: METADATA Group and Map Scale Features

Extended design conversation covering:

### Decisions Made

- **Ground resolution source:** Derived from `elevationCalibration.mapWidth / heightmap.width`. Already in the store; no new user input needed. When `mapWidth` is null, ground resolution and all scale features are unavailable.
- **PNG DPI embedding:** Deferred to open-items doc. Requires post-processing the PNG blob to inject a `pHYs` chunk.
- **Scale at export assumption:** Deferred. Scale calculation assumes 1:1 heightmap pixel export. Added to open-items doc.
- **Scale bar size:** Pick a round real-world distance that comes out ~2.5 cm on the printed output (using `niceBarDistance` ŌĆö rounds to nearest 1/2/5 ├Ś 10Ōü┐). No fixed 1-inch or 3-cm user choice needed.
- **Scale bar units:** Metric vs. Imperial (user toggle). "Imperial" preferred over "US customary" as the UI label.
- **Scale bar position:** Outside the legend box, grouped at the same horizontal alignment, user selects above or below.
- **Scale ratio in legend:** Shown as an italic header text row at the top of the legend box (before symbol items).
- **`calToMeters` sharing:** Move from private function in useStore to exported function in types/index.ts.

After the initial METADATA implementation, the user requested 4 scale bar styles with full styling controls. Decisions:

- **4 styles:** `line` (ticks), `banded` (solid alternating fill), `open` (alternating c2/transparent with c1 stroke), `classic` (two-row checkerboard: top row c1/c2, bottom row c2/c1)

- **Sub-labels:** classic style can show a second row of labels in the alternate unit system
- **Scale ratio styling:** per-field font size, color, bold, italic (previously always used legend defaults)
- **`niceBarDistance` moved to `types/index.ts`** so both MapCanvas and ParameterPanel can import it

## Feature: Three-Way Hillshade View Control

### Request

Replace the "Overlay only" boolean Switch in the HILLSHADE section header with three modes:
- **Combined** ŌĆö hillshade image + all SVG overlays (previous default)
- **Hillshade only** ŌĆö hillshade image only, all map content overlays hidden
- **Overlay only** ŌĆö white background + all SVG overlays (previous `overlayOnly = true`)

Frame elements (frame border, margins, title, compass, legend, measure bar) remain visible in all three modes, since the frame can be toggled independently.

### State Change

`overlayOnly: boolean` ŌåÆ `hillshadeView: 'combined' | 'hillshade-only' | 'overlay-only'`

- `types/index.ts`: field renamed in `ProjectState`
- `store/useStore.ts`: `setHillshadeView` action; version 4 ŌåÆ 5; migration converts `overlayOnly: true` ŌåÆ `'overlay-only'`, `false` ŌåÆ `'combined'`; old key deleted from persisted state

### `MapCanvas.tsx` Changes

`const showOverlays = hillshadeView !== 'hillshade-only'` gates:
- Contour SVG (`id="contour-svg"`)
- **Annotation SVG** (`id="annotation-svg"`) ŌĆö this is a separate element from the contour SVG; contains roads, elevation flags, slope arrows, ruggedness flags, swamp markers, buildings, POIs, and curved labels (zOrder < 75)
- Grid canvas
- High-z curved labels SVG (zOrder Ōēź 75)

Frame elements (title, compass rose, legend overlay, measure bar, frame border) are **not** gated ŌĆö they always render when enabled.

### Bug During Implementation

First attempt gated frame elements with `showOverlays` and missed the annotation SVG entirely (it has a different condition from the contour SVG: `{heightmap && (` vs `{contourState && heightmap && !hillshadeGenerating && (`). Result: markers/labels/roads still showed in hillshade-only mode, and the frame disappeared. Fixed by:
1. Adding `showOverlays &&` to the annotation SVG
2. Removing `showOverlays &&` from all five frame element conditions

- `overlayOnly` / `setOverlayOnly` ŌåÆ `hillshadeView` / `setHillshadeView`
- Switch in HILLSHADE header removed; replaced with `SegmentedControl fullWidth` at top of the Hillshade Collapse with three options
