## Fix: Flip / Reverse Side Rendering Logic

### Problem

The rendering logic was wired backwards:
- `side === 'right'` drove path reversal, which both changed sides AND inverted text orientation
- `flip === true` drove `dominantBaseline="hanging"`, which only changed sides (no orientation change)

Result: the two toggles together gave all four variants, but which toggle did which thing was unclear.

### Fix (two `replace_all` edits to `MapCanvas.tsx`)

**Path reversal — now driven by `flip`:**
```
const pts = label.flip ? [...label.points].reverse() : label.points
```

**`dominantBaseline` — XOR logic so "Reverse Side" only changes side:**
```
dominantBaseline={(label.side === 'right') !== label.flip ? 'hanging' : undefined}
```

**Verified 2×2 behavior matrix:**
| side | flip | Result |
|------|------|--------|
| left | false | above path, normal reading direction |
| right | false | below path, normal reading direction |
| left | true | above path, upside-down orientation |
| right | true | below path, upside-down orientation |

## Design Discussion: METADATA Group and Map Scale Features

Extended design conversation covering:

### Decisions Made

- **Ground resolution source:** Derived from `elevationCalibration.mapWidth / heightmap.width`. Already in the store; no new user input needed. When `mapWidth` is null, ground resolution and all scale features are unavailable.
- **PNG DPI embedding:** Deferred to open-items doc. Requires post-processing the PNG blob to inject a `pHYs` chunk.
- **Scale at export assumption:** Deferred. Scale calculation assumes 1:1 heightmap pixel export. Added to open-items doc.
- **Scale bar size:** Pick a round real-world distance that comes out ~2.5 cm on the printed output (using `niceBarDistance` — rounds to nearest 1/2/5 × 10ⁿ). No fixed 1-inch or 3-cm user choice needed.
- **Scale bar units:** Metric vs. Imperial (user toggle). "Imperial" preferred over "US customary" as the UI label.
- **Scale bar position:** Outside the legend box, grouped at the same horizontal alignment, user selects above or below.
- **Scale ratio in legend:** Shown as an italic header text row at the top of the legend box (before symbol items).
- **`calToMeters` sharing:** Move from private function in useStore to exported function in types/index.ts.

## Feature: Hydrological Water Feature Detection

### Request

Build an algorithm that scans the heightmap to find lakes, streams, and rivers with noise controls, adjustable detection parameters, and full styling/label support for detected features.

### Architecture

**`src/renderer/src/utils/hydrology.ts`** — new file containing the full pipeline:
- **MinHeap** class (binary heap for priority-flood)
- **Priority-Flood** (Barnes et al. 2014): fills all depressions by seeding border cells into a min-heap, propagating minimum surface elevation inward
- **D8 flow direction**: assigns each cell the steepest-descent direction to one of 8 neighbors
- **Flow accumulation** (BFS topological sort): each cell's accumulation = sum of all cells that drain through it
- **Strahler order** (stream subgraph BFS): headwaters = 1; at each junction, if ≥2 tributaries have equal max order, order = max+1; otherwise order = max
- **Segment tracing**: processes cells in topological order, groups consecutive same-Strahler-order cells into polylines, breaks at junctions; applies Douglas-Peucker simplification
- **Lake detection**: runs d3-contour marching squares on a depression mask (`filled[i] - original[i] > minDepth`); filters by minimum area; extracts polygon, surface elevation, depth
- **Label auto-placement**: lakes → 3 points from upper shoreline arc; rivers → 3 points at 30/50/70% along main stem

### Data Types (`types/index.ts`)

```typescript
WaterLake   — polygon, areaPx, surfaceElevNorm, depthNorm, color, opacity, label + label styling, labelPoints
WaterRiver  — systemId/Rank, segments[{points, strahlerOrder}], maxAccumulation, color, opacity, strokeWidth, label + label styling, labelPoints
WaterDetectionParams — minDepthPct (%), minAreaPx, accumulationPct (%), maxRiverSystems
```

Defaults: `minDepthPct: 2`, `minAreaPx: 20`, `accumulationPct: 0.5`, `maxRiverSystems: 5`.

### Store (`useStore.ts`)

Added `waterLakes[]`, `waterRivers[]`, `waterLakesVisible`, `waterRiversVisible`, `waterDetectionParams`, `selectedWaterLakeId`, `selectedWaterRiverId`, `waterDetecting` to `ProjectState` and `initialState`. Full CRUD actions. Persisted in `partialize`; deep-merged via `merge`.

### Rendering (`MapCanvas.tsx`)

- **Lakes**: filled polygon at `lake.color / lake.opacity`; clickable to select
- **Rivers**: each segment drawn with `strokeWidth * min(strahlerOrder, 4) / max(maxOrd, 4)`; clickable
- **Selection — lakes**: white glow outline + dashed blue stroke + orange locator pin (downward-pointing triangle with white dot) at polygon centroid
- **Selection — rivers**: white glow + cyan highlight on all segments + 4 orange locator pins at start/40%/70%/end of network
- **Pin sizing**: `svgPinScale = heightmap.width / innerMapSize.w` converts target screen pixels → SVG units, so pins appear at a fixed physical size (~28px tall) regardless of zoom
- **Labels**: rendered as `<textPath>` along catmull-rom curve from `labelPoints`, same system as CurvedLabels

### Parameter Panel (`ParameterPanel.tsx`)

**WATER FEATURES** section (collapsible, at bottom before METADATA):
- **Detect** button: runs detection async (50ms yield to show loading state); shows "Re-detect" when features exist
- **Confirmation modal**: warns "replace X lakes and Y rivers" before re-detecting
- **Detection params** accordion: min depth %, min area px, accumulation %, max river systems
- **Lakes list** (if any): visible switch, delete-all button, per-lake collapsible panel (color, opacity, label text + font/size/color/bold/italic when text is set, delete button)
- **Rivers list** (if any): same structure + base stroke width control
