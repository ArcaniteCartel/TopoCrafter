## Windows Build Fix (winCodeSign + NSIS)

Continued from previous session where `npm run build:win` was failing due to the
`winCodeSign-2.6.0.7z` archive containing macOS symlinks that Windows cannot create
without elevated privileges.

**Resolution:** User enabled Windows Developer Mode (Settings ‚Üí System ‚Üí For Developers).
This grants the `SeCreateSymbolicLinkPrivilege` right to normal user processes. The build
ran, winCodeSign extracted successfully, and the cache is now populated for future builds.
Developer Mode can be turned off again after the first successful build.

**NSIS bug fixed during build:** The `installer.nsh` macro used `MB_ICONWARNING` in a
`MessageBox` call. NSIS's parser only recognises `MB_ICONEXCLAMATION` (the Win32 name);
`MB_ICONWARNING` is a Windows API alias that NSIS doesn't expose. Fixed and rebuilt.

## EXR Heightmap Support ‚Äî Sharp Replaced with hdrify

### Problem
Sharp 0.34.5 (used in previous session for EXR decoding via the main process) does not
include OpenEXR support in its pre-built Windows binaries. Any attempt to load an EXR
file through `sharp().metadata()` raised `Error: Input file contains unsupported image
format`.

### Investigation
- Confirmed that `@loaders.gl/exr` does not exist on npm (404).
- `openexr-js`, `exr-parser`, `node-openexr` also don't exist.
- User suggested the `hdrify` package.
- `npm info hdrify` confirmed: MIT, pure JS (deps: `buffer`, `fflate`, `jpeg-js`),
  supports EXR + HDR/RGBE, works in Node and browser. No native code.

### Solution
Replaced the entire sharp/IPC pipeline with `hdrify` running directly in the renderer.

**Files changed:**
- `src/renderer/src/utils/heightmap.ts` ‚Äî removed `parseHeightmapExr(filePath)` IPC
  approach; new `parseHeightmapExr(buffer)` calls `readExr(buffer)` from hdrify, receives
  RGBA `Float32Array`, computes BT.709 luminance per pixel, normalises to `[0, 1]`.
- `src/main/index.ts` ‚Äî removed `import sharp` and the `heightmap:decodeExr` IPC handler.
- `src/preload/index.ts` ‚Äî removed `decodeExr` from the context bridge API.
- `src/renderer/src/types/index.ts` ‚Äî removed `decodeExr` from `window.electronAPI`
  declaration.
- `package.json` ‚Äî removed `sharp`, added `hdrify ^0.12.1`.
- `build/installer.nsh` ‚Äî deleted (VC++ redistributable was only needed because sharp
  requires the MSVC runtime; hdrify has no native deps).
- `package.json` (build config) ‚Äî removed `"include": "build/installer.nsh"` from NSIS
  section.

**Commit:** `7af9684`

## Sticky Tabs ‚Äî Prevent Scrolling Off Screen

**Problem:** When the map image was scrolled vertically, the Terrain/Hillshade tab bar
would scroll off the top of the viewport.

**Root cause (final analysis):** `height: '100%'` on `AppShell.Main` referenced the
parent AppShell root's `min-height: 100dvh`, giving Main a computed height of `100dvh`.
But Main starts at y=52 (after the sticky header in the flex column), so it extended to
y=52+100dvh ‚Äî below the viewport ‚Äî causing body-level scroll instead of canvas-level
scroll. The tabs scrolled away with the body.

**Attempts:**
1. `overflow: 'hidden'` on `AppShell.Main` ‚Äî no effect because Main's height still
   overflowed (height > viewport, so there was still a scroll container above it).
2. `position: 'sticky', top: 52, zIndex: 100` on the `Tabs` component ‚Äî partially worked
   (tabs stayed visible) but caused the horizontal scrollbar issue described below.

**Final fix** (`src/renderer/src/App.tsx`):
- Removed `height: '100%'` from `AppShell.Main` style ‚Äî now `flex: 1` from Mantine's
  CSS correctly sizes Main to `100dvh ‚àí 52px`.
- Added `overflow: 'hidden'` to `AppShell.Main` ‚Äî body no longer scrolls.
- Removed `position: sticky` from the Tabs (no longer needed; tabs are always visible
  because body scroll is gone).

  inserted into the right-side group before the Export menu.

  - Slider internal range: ‚àí100 to 100, step 5.

### MapCanvas (`src/renderer/src/components/MapCanvas/MapCanvas.tsx`)

## Horizontal Scrollbar ‚Äî Fixed at Viewport Bottom

**Problem:** When the image was zoomed in (horizontal overflow), scrolling the image
vertically caused the horizontal scrollbar to disappear off the bottom of the screen.

**Root cause:** Mantine's AppShell uses a body-scroll paradigm ‚Äî the root has
`min-height: 100dvh`, not `height: 100dvh`. No ancestor of MapCanvas had a definitively
bounded height, so `flex: 1, minHeight: 0, overflow: auto` on the MapCanvas outer div had
nothing to constrain it against. The browser placed both scrollbars at the body level,
not inside the canvas region.

**Failed attempts:**
1. `overflow: hidden` on `AppShell.Main` with flex layout ‚Äî body still scrolled in some
   configurations.
2. `minHeight: 0` on the MapCanvas outer div ‚Äî correct pattern but ineffective when the
   flex parent itself is unbounded.
3. `overflow: hidden` on `html, body, #root` ‚Äî clipped all content; no scrollbars at all.

**Final fix** (`src/renderer/src/App.tsx`):
Replaced the flex layout on `AppShell.Main` with a `position: fixed` inner div that uses
Mantine CSS variables to anchor exactly to the visible viewport region:

```tsx
<AppShell.Main>
  <div style={{
    position: 'fixed',
    top: 'var(--app-shell-header-height, 52px)',
    left: 'var(--app-shell-navbar-width, 300px)',
    right: 0,
    bottom: 0,
    display: 'flex',
    flexDirection: 'column',
    overflow: 'hidden',
  }}>
    {showTabs && <Tabs style={{ flexShrink: 0 }} ... />}
    <MapCanvas />  {/* flex: 1, minHeight: 0, overflow: auto */}
  </div>
</AppShell.Main>
```

This definitively bounds the content area to the visible viewport regardless of Mantine's
body-scroll architecture. MapCanvas's `overflow: auto` now has a true bounded parent, so
both scrollbars are anchored within the canvas area and never scroll off screen.

Also added `minHeight: 0` to the MapCanvas outer div as part of the correct
`flex: 1, minHeight: 0, overflow: auto` bounded-scroll-container pattern.
