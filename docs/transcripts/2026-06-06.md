# TopoCrafter Session — 2026-06-06

## Footpath & Trail Road Types

### Feature Overview

Two new single-line road variants added alongside the existing dual-lane dirt/gravel/paved types:

- **Footpath** — dotted single center line (`strokeDasharray: ${sw} ${sw*3}`), no label support, brown default color (`#8B6914`)
- **Trail** — dot-dash single center line (`strokeDasharray: ${sw} ${sw*2} ${sw*5} ${sw*2}`), supports named labels offset above the line, darker brown default (`#5C4A2A`)

Same spline/drag/delete mechanics as existing road types. Hit areas still use `trackWidth` for a usable click target.

### Single-Line vs Dual-Lane Rendering

An `isSingleLine` flag gates the render path:

```typescript
const isSingleLine = road.type === 'footpath' || road.type === 'trail'
```

- **Dual-lane** (dirt/gravel/paved): two offset `<path>` elements using `catmullRomOffsetPath` at `±half` track width
- **Single-line** (footpath/trail): one center `<path>` element at stroke weight `sw`

SVG mask knockout width is proportional to the visual element:

```typescript
const maskStrokeW = isSingleLine
  ? road.strokeWeight * 4
  : road.trackWidth + road.strokeWeight * 2 + road.trackWidth * 0.3
```

### Trail Label Offset

Footpath never renders a label. Trail labels use SVG `dy` on the `<text>` element containing `<textPath>` to shift text perpendicular to the curve, lifting it above the stroke:

```tsx
{road.label && road.type !== 'footpath' && (
  <text fontSize={tw * 0.7} fontFamily={style.labelFont} fill={road.color}
    dominantBaseline="middle" textAnchor="middle"
    dy={road.type === 'trail' ? -(tw * 0.7 * 0.8) : 0}>
    <textPath href={`#road-center-${road.id}`} startOffset="50%">
      {road.label}
    </textPath>
  </text>
)}
```

## Black Screen Crash Fix

### Symptom

After the per-type legend changes, the app would briefly show the UI on startup then turn entirely black.

### Root Cause

In `MapCanvas.tsx`'s `LegendOverlay`, the `items.map` callback destructured only `{ type, label }` from each item — not `color`. The per-type road branches (`road-dirt`, `road-gravel`, etc.) referenced `color` as a bare variable. The only `color` local variable was `const color = swampMarkerDefaults.color` declared inside the `swamp` else-if block, which is block-scoped and not accessible in sibling branches. At runtime, accessing `color` in a road-type branch threw `ReferenceError: color is not defined`, crashing the React component. TypeScript's control flow analysis did not catch this.

### Fix

Changed the map callback parameter to destructure `color`:

```tsx
// Before (buggy):
{items.map(({ type, label }, i) => {

// After (fixed):
{items.map(({ type, label, color }, i) => {
```

Removed the now-redundant `const color = swampMarkerDefaults.color` from the swamp branch — the destructured `color` already equals `swampMarkerDefaults.color` since that value is stored in each item object.

TypeScript clean (`npx tsc --noEmit` no output).

## Road Tool Elevation Readout

### Feature Overview

When the road tool is active, the current elevation at the crosshair position is displayed beside the cursor as it moves over the map. The label appears to the upper-right of the cursor, in the current road type's color, and disappears when the cursor leaves the map.

The label only renders when elevation calibration data is available (`realMin` / `realMax` set). It includes the configured unit abbreviation (ft, m, or custom), matching the rest of the app.

### Implementation

**`roadHoverPt` state** extended from `{ x, y }` to `{ x, y; elevation?: number }`:

```typescript
const [roadHoverPt, setRoadHoverPt] = useState<{ x: number; y: number; elevation?: number } | null>(null)
```

**`handleSvgMouseMove`** — when `mapTool === 'road'`, calls `computeElevationAt` and stores result alongside the position:

```typescript
if (mapTool === 'road') {
  const pt = getSvgPoint(e.clientX, e.clientY)
  if (pt) {
    const elevation = computeElevationAt(pt.x, pt.y)
    setRoadHoverPt({ x: pt.x, y: pt.y, elevation: elevation ?? undefined })
  }
  return
}
```

**Render block** added to the annotation SVG between the road drawing preview and the existing hover preview. Uses the current road type's color and the calibration unit abbreviation:

```tsx
{mapTool === 'road' && roadHoverPt && roadHoverPt.elevation !== undefined && (() => {
  const color = roadDefaults.type === 'dirt' ? roadDefaults.dirtColor : ...
  const unitAbbr = elevationCalibration.unitType === 'feet' ? 'ft' : ...
  return (
    <text x={roadHoverPt.x + s * 0.9} y={roadHoverPt.y - s * 0.3}
      fontSize={s} fontFamily={style.labelFont} fill={color}
      dominantBaseline="middle" style={{ pointerEvents: 'none' }} opacity={0.85}>
      {roadHoverPt.elevation}{unitAbbr}
    </text>
  )
})()}
```

TypeScript clean (`npx tsc --noEmit` no output).

## Urban Areas & Settlements — Building Placement Tool

### Feature Overview

New `'building'` map tool that places historical building footprints on the map as filled SVG shapes, automatically scaled from real-world meters to map pixels using the elevation calibration ground resolution.

- Click to place a building at the cursor position
- Hover preview shows the shape before placement
- Buildings are draggable and deletable (Delete key when selected)
- 15 culture/period groups, each with 4–8 building templates, spanning Neolithic Europe through the Renaissance
- 8 distinct SVG plan shapes covering all historical building types

### Shape Vocabulary

| Shape | Description |
|---|---|
| `rectangle` | Simple rect (most common — longhouses, granaries, cottages) |
| `circle` | Ellipse (roundhouses, grain silos, rotundas) |
| `bow-sided` | Rect with long walls bowing outward via quadratic bezier (`bow = hw * 1.14`) |
| `apsidal` | Rect with a semicircular apse at one end (early churches, Mycenaean megara) |
| `courtyard` | Hollow rectangle using SVG `fillRule="evenodd"` (inner void = 60% of outer dims) |
| `L-shape` | 6-point polygon — upper-right quadrant removed |
| `U-shape` | 8-point polygon — bottom notch (notch width = 50% of total) |
| `octagon` | 8-point polygon clipping corners with `cut = 0.2929` (= 1 − 1/√2) |

### Ground Resolution / Pixels-per-Meter

```typescript
const pixelsPerMeter = (() => {
  if (!heightmap || !elevationCalibration.mapWidth || elevationCalibration.mapWidth <= 0)
    return heightmap ? heightmap.width * 0.01 : 1
  const metersPerUnit = cal.unitType === 'feet' ? 0.3048
    : cal.unitType === 'meters' ? 1
    : (cal.customRatio ?? 1) * (cal.customBase === 'feet' ? 0.3048 : 1)
  return heightmap.width / (cal.mapWidth * metersPerUnit)
})()
```

Falls back to `heightmap.width * 0.01` when uncalibrated so buildings remain visible at a reasonable size.
