## Fix: Real-World Contour Interval Field Not Editable

**Problem:** The "Interval (ft)" TextInput in the Parameter Panel accepted focus but blocked all keyboard input.

**Root cause:** The field had `disabled={!calReady}` — Mantine's disabled input blocks the underlying `<input>` element even though the wrapper div appears focusable.

**Fix:** Removed the `disabled` prop entirely from the TextInput. Added a `calReady` guard inside `handleIntervalInput` so store updates are skipped when calibration isn't ready, but the user can still type freely.

## Architecture Change: Decouple Controls from Recalculation

**Problem:** Every parameter change auto-triggered contour recalculation (via `useMemo`) or hillshade regeneration (via a 300ms debounced `useEffect`), causing performance issues and interfering with user input.

**Design decision (Q&A with user):**
- Style controls (colors, widths, opacity, label toggle) remain live/instant — no recalc needed
- Hillshade controls + contour controls require explicit user action to apply
- "Rerun Hillshading" button hidden when not in hillshade mode
- Fresh heightmap load still auto-runs the full pipeline

**Implementation:**

### Store additions (`src/renderer/src/store/useStore.ts`, `src/renderer/src/types/index.ts`)
- Added `hillshadeDirty: boolean`, `contoursDirty: boolean`, `hillshadeVersion: number`, `contoursVersion: number` to `ProjectState`
- `updateHillshadeParams` now sets `hillshadeDirty: true`
- `updateParameters`, `updateElevationCalibration`, `setElevationUnits` now set `contoursDirty: true`
- `setTerrainIsHillshade(true)` increments `hillshadeVersion` (triggers auto-run on load)
- `setHeightmap` increments `contoursVersion` (triggers auto-run on load)
- New actions: `triggerHillshade()` (increments version, clears dirty), `triggerContours()` (same), `setHillshadeDirty()`, `setContoursDirty()`

### Hook: `src/renderer/src/hooks/useHillshade.ts`
- Removed `hillshadeParams` from `useEffect` deps — no longer auto-fires on param changes
- Added `hillshadeVersion` to deps — fires only when explicitly triggered or on load
- Added `hillshadeParamsRef` so the async callback always reads latest params without stale closure

### MapCanvas: `src/renderer/src/components/MapCanvas/MapCanvas.tsx`
- Replaced `useMemo` contour generation with `useState` + `useEffect([heightmap, contoursVersion])`
- Added `parametersRef` so the effect always reads the latest parameters when triggered
- Contours only recompute on new heightmap load or explicit "Recalculate Contours" click

### FilePanel: `src/renderer/src/components/FilePanel/FilePanel.tsx`
- Added two new Paper-bordered button sections below the existing load buttons, styled identically:
  - **"Rerun Hillshading"** — visible only in hillshade mode; enabled when `hillshadeDirty`; shows loading spinner while generating; disables after run
  - **"Recalculate Contours"** — always visible; enabled when `contoursDirty && heightmap`; synchronous, no spinner needed
- After heightmap load: explicitly calls `setHillshadeDirty(false)` and `setContoursDirty(false)` to keep buttons dormant after the auto-triggered run

## Fix: Contour Label Values Incorrect

**Problem:** With min=0 ft, max=1500 ft, interval=50 ft, the first major contour (every 5th) was labeled 77 instead of 250.

**Root cause:** Label formula treated the normalized threshold as a 0→1 fraction of the full range:
```ts
// Wrong
const elev = Math.round(realMin + threshold * (realMax - realMin))
```
But thresholds run from `parameters.minElevation` to `parameters.maxElevation` (e.g., 0→0.3063), not 0→1. So `threshold=0.05105` produced `0.05105 × 1500 = 76.5 ≈ 77` instead of the correct 250.

**Fix:** Correct linear interpolation using the actual normalized span:
```ts
const elev = Math.round(realMin + (threshold - normMin) / normSpan * (realMax - realMin))
```

**File:** `src/renderer/src/components/MapCanvas/MapCanvas.tsx`

## Fix: Label Off-By-One Rounding (e.g., 749 instead of 750)

**Problem:** With interval=75 ft, major contour labels showed 749, 1124, 1499 instead of 750, 1125, 1500.

**Root cause:** Floating-point accumulation in the threshold array (generated by repeated `+= interval` in a loop) caused values like threshold[10] = `0.149999...` instead of exactly `0.15`. Dividing by `normSpan` and multiplying by `realSpan` then landed just below the integer, rounding down.

**Fix:** Bypass the threshold entirely. Use the contour index `i` and the integer `realInterval` for exact arithmetic:
```ts
const elev = (realInterval !== null)
  ? realMin + i * realInterval          // exact integers always
  : Math.round(realMin + (threshold - normMin) / normSpan * (realMax - realMin))  // fallback
```
Since `realMin` (0) and `realInterval` (75) are both integers, `0 + i × 75` is always exact.

**File:** `src/renderer/src/components/MapCanvas/MapCanvas.tsx`

## Fix: Recalculate Contours Spinner Not Showing

**Problem:** Clicking "Recalculate Contours" set `contoursGenerating: true` but no spinner was ever visible — computation completed before the browser could paint the loading state.

**Root cause:** `generateContours` is synchronous. Even with `setTimeout(0)` or a double `requestAnimationFrame`, the JS engine completes the computation within the same paint frame, so the spinner never renders.

**Fix (attempt 1 — setTimeout(0)):** Did not work. Computation still finished before paint.

**Fix (attempt 2 — double rAF):** Button blanked out but spinner didn't animate visibly.

**Fix (attempt 3 — 300ms setTimeout):** Matching the hillshade pattern (which already used a 300ms debounce), the contour effect now waits 300ms before starting computation. This gives the browser a full third of a second to render and animate the Mantine `Loader` spinner. Works reliably.

**Files:** `src/renderer/src/components/MapCanvas/MapCanvas.tsx`

## Fix: Custom Unit Conversion Incorrect

**Problem:** When switching to "Custom" unit type, the store immediately ran unit conversion using `customRatio: 1` (the default), wrongly treating 1 custom unit = 1 foot/meter and producing no visible change to the real-world values.

**Root cause:** The `setElevationUnits` conversion path ran for all unit type changes, including `→ custom`, even though the custom ratio hadn't been entered yet.

**Fix:** Added an explicit `if (newType === 'custom')` branch that clears `realMin`, `realMax`, and `realInterval` to `null` rather than attempting conversion. The user then enters fresh values in their custom unit after defining the ratio.

**File:** `src/renderer/src/store/useStore.ts`

## Fix: Labels Updating Live Before Recalculate

**Problem:** Changing any calibration parameter (unit type, real-world min/max, interval) immediately updated the contour labels on screen, even though the contour lines themselves hadn't been recomputed. This produced inconsistent labels that didn't match the lines they annotated.

**Root cause:** Labels read directly from live Zustand store state — so any `updateElevationCalibration` call instantly changed the displayed elevation values.

**Fix:** Introduced a `ContourState` interface that snapshots calibration values at compute time:
```ts
interface ContourState {
  contourSet: ContourSet
  realMin: number | null
  realMax: number | null
  realInterval: number | null
  minElevation: number
  maxElevation: number
}
```
The `useEffect` captures `elevationCalibrationRef.current` and `parametersRef.current` inside the 300ms setTimeout, stores the snapshot in `useState<ContourState>`, and labels read from the snapshot — not from the live store. Labels now only update when "Recalculate Contours" is clicked.

**File:** `src/renderer/src/components/MapCanvas/MapCanvas.tsx`

## Fix: Path Smoothing Slider Had No Effect

**Problem:** Moving the Path Smoothing slider and clicking "Recalculate Contours" produced identical contour lines regardless of the slider value.

**Root cause:** `generateContours` destructured `interval, minElevation, maxElevation, majorEvery` from `params` but never destructured `smoothing`. The value was silently ignored.

**Fix:** Added a `boxBlur` function (separable two-pass box blur over the `Float32Array` elevation data) and wired it into `generateContours`:
```ts
const radius = Math.round(smoothing * 8)  // 0–8 pixel blur radius
const effectiveData = radius > 0 ? boxBlur(data, width, height, radius) : data
const paths = generator(Array.from(effectiveData))
```
Blurring the terrain data before d3-contour runs produces naturally smooth, rounded contour lines at higher smoothing values.

**File:** `src/renderer/src/utils/contour.ts`
